# Codex Input Preview Design

## Summary

Add a standalone `codex-input-preview` Omniskills workflow that produces a
faithful Codex-style composer preview as a 1200 × 675 PNG. Publish one real
generated example on the workflow's landing detail page together with copyable
install and invocation commands.

The preview is a simulated UI artifact. It does not open a Codex session, call a
model, or claim to show live product output.

## User and Job

The target user is an Omniskills author or adopter who wants a clear,
shareable example of a Codex prompt configuration. The generated image should
communicate the prompt, selected model, and reasoning effort without requiring
the viewer to interpret configuration JSON or terminal output.

## Approved Usage Contract

Install the workflow:

```text
npx omniskill@latest install codex-input-preview
```

Invoke the installed skill with natural language:

```text
$codex-input-preview Draw “Help me announce that I’m joining the Codex team!” using GPT-5.6 with high effort.
```

The skill extracts and passes three content inputs to its renderer:

- `prompt`: required text rendered in the composer.
- `model`: required free-form display label so future model names work without a
  skill update.
- `effort`: required enum: `low`, `medium`, `high`, or `xhigh`.

The default output is `./codex-input-preview.png`. A user may request another
path, but the path must end in `.png`.

## Scope

- One independently installable workflow at
  `examples/workflows/codex-input-preview/`.
- One local entry skill named `codex-input-preview`.
- One bundled Node renderer script using only Node built-ins and an installed
  Chrome-family browser.
- A faithful light Codex composer on a 1200 × 675 white canvas.
- A checked-in PNG generated by the renderer at
  `landing/public/examples/codex-input-preview.png`.
- A workflow card and detail-page example with copyable install and invocation
  commands.
- Clear validation, cleanup, and browser-not-found errors.

## Non-goals

- Editable SVG or another exposed source format.
- Themes, custom canvas sizes, or additional output formats.
- An interactive landing-page image editor.
- Live Codex execution, model calls, or network access.
- Pixel-identical reproduction of proprietary Codex implementation details.
- Decorative motion or a new landing-page animation system.

## Workflow Bundle

The bundle follows the existing one-entry-skill workflow pattern:

```text
examples/workflows/codex-input-preview/
  README.md
  workflow.json
  workflow.lock.json
  skills/codex-input-preview/
    SKILL.md
    agents/openai.yaml
    scripts/render-preview.mjs
```

`workflow.json` declares schema `0.1`, version `0.1.0`, one entry skill, and one
step that creates the PNG. The README documents public installation plus local
authoring commands. The lock file is produced by `omniskill lock` after all
skill resources are final.

The skill interprets the user's natural-language request, invokes the bundled
renderer with explicit arguments, and reports the final absolute output path.
It does not recreate the renderer inline or expose a temporary HTML file.

## Renderer Interface

The renderer accepts explicit flags:

```text
node scripts/render-preview.mjs \
  --prompt "Help me announce that I’m joining the Codex team!" \
  --model "GPT-5.6" \
  --effort high \
  --output ./codex-input-preview.png
```

It performs these steps in order:

1. Parse flags and reject missing, duplicate, or unknown arguments.
2. Validate `effort`, the `.png` output extension, and a non-empty prompt and
   model.
3. Locate a supported Chrome, Chromium, or Edge executable through a small
   platform-specific candidate list. Controlled environments may set the
   `CODEX_INPUT_PREVIEW_BROWSER` environment variable; this is an operational
   override, not a fourth content input.
4. Create a uniquely named temporary HTML file outside the destination.
5. Render the composer with inline HTML and CSS and check whether the prompt
   fits within four lines. The renderer may reduce the prompt font through a
   fixed small set of sizes before declaring overflow.
6. Ask the browser to capture exactly 1200 × 675 CSS pixels at device scale 1.
7. Validate the PNG signature and dimensions before replacing the destination.
8. Delete temporary files in a `finally` path whether the run succeeds or
   fails.

The destination is written atomically: capture to a sibling temporary PNG,
validate it, then rename it into place. A failed render must not leave a new or
partially written destination, and must preserve an existing destination.

## Visual Design

The generated PNG uses the approved clean-composer direction:

- White 16:9 canvas with the composer centered in the lower-middle visual band.
- White rounded composer with a subtle neutral border and soft shadow.
- Prompt in a system UI font with high-contrast near-black text.
- Fixed decorative controls for add context, microphone, and submit.
- Model and effort rendered together in the composer footer.
- No installation command, skill invocation, disclosure, logo, or browser frame
  inside the PNG.

The design is Codex-inspired and recognizable, but uses simple original shapes
and system icons rather than copied product assets.

## Text Fit and Errors

The prompt may occupy at most four lines. The browser template tries approved
font sizes from largest to smallest and records a machine-readable fit status.
If the smallest size still overflows, the renderer exits with a message asking
the user to shorten the prompt. It never crops or silently truncates text.

Other failures use actionable messages:

- Missing browser: name the supported browser families and the explicit-path
  option.
- Invalid effort: list the four accepted values.
- Invalid output: require a writable `.png` path.
- Browser failure: include the exit status without dumping unrelated browser
  logs.
- Invalid capture: reject a missing, malformed, or incorrectly sized PNG.

## Landing Integration

Extend the landing content contract with an optional usage example containing:

- PNG source path.
- Descriptive alt text that states the prompt, model, and effort.
- Copyable invocation text.
- A short disclosure caption.

Only the `codex-input-preview` workflow sets this field initially. Durable copy
and example data stay in `landing/lib/landing-content.ts`; the route remains
presentational.

Generalize the existing install-only copy control into a `CopyableCommand`
component with configurable visible label and accessible label. Use it for both
the existing install command and the new skill invocation.

On the workflow detail route, render the optional example below the entry-skill
panel in the main column. The section order is:

1. Install command.
2. Entry skill.
3. Example input heading and generated PNG.
4. Copyable invocation.
5. Caption: simulated Codex composer preview, not a live session.

The workflow list card remains compact and links to the detail route; it does
not embed the large preview.

## Responsive and Accessible Behavior

- Scale the PNG to the available width while preserving 16:9 proportions.
- Prevent horizontal overflow at 320px and retain the existing two-column
  detail layout at large breakpoints.
- Use useful alt text rather than repeating the disclosure caption.
- Preserve keyboard operation and a visible focus state on copy controls.
- Announce copied state through button text; do not rely on motion or color.
- Add no new decorative animation.

## Verification

### Renderer

- Focused tests cover argument parsing, effort validation, output-extension
  validation, browser-not-found behavior, overflow signaling, and cleanup.
- A smoke run generates the landing example through the bundled renderer.
- Tests parse the PNG header directly to assert its signature and exact
  1200 × 675 dimensions without adding an image dependency.
- Visually inspect the checked-in example against the approved faithful-Codex
  direction.

### Workflow

- Add a bundle regression test covering the name, entry skill, single step,
  script resource, and generated example contract.
- Run `rtk bun run dev -- validate examples/workflows/codex-input-preview`.
- Run `rtk bun run dev -- deps examples/workflows/codex-input-preview`.
- Generate and validate `workflow.lock.json` with the repository CLI.

### Landing

- Test the new workflow metadata and optional usage-example content.
- Test static route generation, copyable command labels, image alt text, and
  simulated-preview disclosure.
- Run the landing test, typecheck, and production build.
- Browser-smoke the detail page at 320px, 768px, and 1440px.

### Final Gate

Run `rtk bun run check` from the repository root. Completion requires all checks
to pass and the generated PNG to be present at the content-declared path.

## Risks and Mitigations

- **No Chrome-family browser installed:** fail before rendering with a concise
  installation/path message; do not introduce a native rendering dependency.
- **Browser CLI differences:** keep browser discovery and invocation behind one
  renderer seam and cover its command construction with tests.
- **Reference drift:** treat the preview as a stable Codex-inspired example,
  not a promise to mirror every Codex UI release.
- **Landing contract growth:** keep usage examples optional and represented by
  one small content object rather than branching the page by workflow slug.

## Approval Gates

3. The plan must be reviewed before dispatching the standard-tier `implement`

4. Independent QA remains the release gate after implementation.
